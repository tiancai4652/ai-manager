     AI Manager — 架构设计与实现计划

     Context

     问题: 使用 Claude Code / CodeX
     做产品时，需要多轮对话才能完成任务，无法一次性交代需求后让 AI
     自主执行到底。

     方案: 构建一个 "AI 任务监督者"——通过 node-pty 控制终端中的编码 AI（Claude
     Code / CodeX），自动分析输出、判断状态、注入指令，代替用户完成多轮监督。

     核心原理: 用 node-pty 创建伪终端会话，让编码 AI
     以为在和真人交互；同时用另一个 LLM（Claude
     API）作为"大脑"，实时分析终端输出、判断下一步动作、生成要发送的指令。

     核心组件设计

     1. PTY 终端会话 (terminal/pty-session.ts)

     // 核心职责: 管理一个 node-pty 伪终端实例
     interface IPtySession {
       spawn(command: string, args: string[], cwd: string): void;
       write(input: string): void;          // 注入输入
       onOutput(callback: (data: string) => void): void;  // 实时输出
       getScreenSnapshot(): string;          // 当前屏幕内容
       isAlive(): boolean;
       kill(): void;
     }

     关键细节:
     - 使用 node-pty 的 spawn 创建伪终端
     - 输出流通过 onData 实时接收
     - 用 strip-ansi 清洗输出给 LLM 分析
     - 保留原始 ANSI 输出用于显示
     - 维护一个滑动窗口的输出缓冲区

     2. 输出分析器 (brain/output-analyzer.ts)

     // 核心职责: 让 LLM 分析终端输出, 判断当前状态
     interface OutputAnalysis {
       state: 'idle' | 'working' | 'waiting_input' | 'error' | 'completed';
       summary: string;             // 终端正在做什么
       detectedIssues: string[];    // 发现的问题
       needsIntervention: boolean;  // 是否需要人工介入
       suggestedAction?: string;    // 建议的下一步
     }

     分析方法: 将最近 N 行终端输出（去 ANSI）发送给 Claude API，附带分析
     prompt，返回结构化判断。

     3. 指令生成器 (brain/instruction-generator.ts)

     // 核心职责: 根据 Task 描述 + 当前终端状态, 生成要发送给编码 AI 的指令
     interface Instruction {
       content: string;             // 要输入到终端的内容
       waitFor?: number;            // 发送后等待时间 (ms)
       expectPattern?: string;      // 期望看到的输出模式
     }

     4. 质量评审器 (brain/quality-reviewer.ts)

     // 核心职责: Task 完成后, 检查是否真的做好了
     interface ReviewResult {
       passed: boolean;
       score: number;               // 0-10
       issues: string[];
       suggestedFix?: string;       // 如果不通过, 建议的修复指令
     }

     评审方式:
     - 检查文件是否存在 (fs.existsSync)
     - 运行测试命令, 观察结果
     - LLM 审查代码变更内容
     - 检查是否有明显错误 (lint, type check)

     5. 任务编排器 (core/orchestrator.ts)

     主循环:
     ┌─────────────────────────────────────────────────┐
     │  1. 取出下一个 pending Task                       │
     │  2. 生成初始指令 → 通过 PTY 发送给编码 AI          │
     │  3. 进入监控循环:                                  │
     │     a. 收集终端输出                                │
     │     b. LLM 分析输出状态                            │
     │     c. 根据状态决策:                               │
     │        - working → 继续等待                        │
     │        - waiting_input → 生成回答 → 发送           │
     │        - error → 分析错误 → 生成修复指令 → 发送    │
     │        - idle → 检查是否完成了当前 Task             │
     │  4. Task 完成后, 运行质量评审                       │
     │     - 通过 → 标记完成, 进入下一个 Task              │
     │     - 不通过 → 生成修复指令, 回到步骤 2 (最多 N 次) │
     │  5. 所有 Task 完成 → 通知用户                       │
     └─────────────────────────────────────────────────┘

     6. 数据模型 (models/)

     // 任务状态
     type TaskStatus = 'pending' | 'in_progress' | 'reviewing' | 'completed' |
     'failed' | 'blocked';

     interface Task {
       id: string;
       title: string;
       description: string;
       status: TaskStatus;
       attempts: number;            // 已尝试次数
       maxAttempts: number;         // 最大重试次数
       instructions: string[];      // 发送过的指令历史
       reviewResults: ReviewResult[];
       createdAt: Date;
       updatedAt: Date;
     }

     interface Plan {
       id: string;
       userRequirement: string;     // 用户原始需求
       tasks: Task[];
       currentTaskIndex: number;
       overallStatus: 'planning' | 'executing' | 'completed' | 'failed';
       createdAt: Date;
     }
